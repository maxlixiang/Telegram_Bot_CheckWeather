# AI 编程交接文档

## 1. 任务目标

继续维护一个基于 Python 3.11、python-telegram-bot、SQLite、Docker 的 Telegram Weather Bot。

当前项目目标是作为**单用户个人天气机器人**长期运行，支持：

- 城市管理
    
- 天气查询
    
- 每日自动推送
    
- 自定义推送时间
    
- Docker / VPS 部署运行
    

当前项目已经完成可用版本开发，并已完成本地验收与 VPS 部署。

---

## 2. 当前状态总览

项目当前已处于**可运行、可部署、可继续迭代**状态。

当前已实现核心能力：

- `/help`：查看帮助
    
- `/add 城市名`：添加城市
    
- `/delete 城市名`：删除城市
    
- `/list`：查看城市列表
    
- `/check`：查询当前已保存城市天气
    
- `/start`：开启每日自动天气推送
    
- `/stop`：关闭每日自动天气推送
    
- `/settime HH:MM`：设置每日自动推送时间
    

当前项目特征：

- 单用户模式，基于 `TELEGRAM_USER_ID` 授权
    
- SQLite 持久化，数据库文件为 `data/weather.db`
    
- 自动推送使用 `python-telegram-bot` 的 JobQueue
    
- 天气数据来源于 Open-Meteo（geocoding + forecast）
    
- 已完成 Docker / Compose 整理
    
- 已完成 GitHub 仓库整理

## 8. 风险与注意事项

### 1）不要误改 `build_weather_text()` 的复用地位

当前：

- `/check`
    
- 自动推送
    

都依赖统一的天气文本构造逻辑。  
如果后续分别改两套格式，很容易产生逻辑分叉。

### 2）不要覆盖原始 `city_name`

旧数据懒补全只应补：

- `display_name`
    
- `latitude`
    
- `longitude`
    
- `normalized_key`
    

不应改掉用户最初输入的 `city_name`。

### 3）不要分散 `normalized_key` 规则

当前 `normalized_key` 是城市去重核心。  
后续如果把规则分散到 handler / db / service 多处，维护成本会迅速上升。

### 4）不要回退 `cities` 表结构

当前代码已经依赖以下字段：

- `display_name`
    
- `latitude`
    
- `longitude`
    
- `normalized_key`
    

删除或回退这些字段会破坏第七阶段能力。

### 5）注意 `list_cities()` 的返回类型变化

后续修改时不要再假设它是早期的 `list[str]`。  
当前更可能是基于 `StoredCity` 之类对象返回。

### 6）不要把运行数据重新纳入 Git 或镜像上下文

当前 `.gitignore` 与 `.dockerignore` 已处理：

- `.env`
    
- `data/weather.db`
    
- 本地虚拟环境
    
- Python 缓存
    
- 编辑器配置
    

不要把这些重新纳入版本控制或镜像上下文。

### 7）不要误以为当前项目还处于“功能未完成”阶段

当前项目已经不是“骨架项目”，而是一个已经完成核心功能、已部署成功的个人工具。  
后续应优先做打磨和稳定性增强，而不是重复补基础功能。
